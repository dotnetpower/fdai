# Rule-catalog profiles and collectors

The FDAI rule catalog has three tiers of content:

1. **Hand-authored rules** under [`rule-catalog/catalog/`](../../../rule-catalog/catalog) -
   curated, T0-ready, ship with a real Rego check and a
   PR-native remediation template. Loaded non-recursively by the T0
   engine.
2. **Machine-imported rules** under [`rule-catalog/collected/`](../../../rule-catalog/collected) -
   auto-generated by the collector pipeline from public upstream
   sources (Azure Policy built-in library, kube-bench, ...). Each entry
   ships with `check_logic.kind: expression` + a `reference:` to the
   upstream definition; they are *reference-only* until a fork or
   upstream curator authors real Rego and re-lands the rule under
   `catalog/`.
3. **Profiles** under [`rule-catalog/profiles/`](../../../rule-catalog/profiles) -
   named bundles that reference rule ids from either tier. Profiles
   are how operators / forks activate a curated subset in one step.

This document is the design contract for the profile layer and every
collector shipped by upstream, per the strategic decisions in
[scope-expansion.md § 3](../fork-and-sequencing/scope-expansion.md).

## 1. Profiles

Design contract: [scope-expansion.md § 3](../fork-and-sequencing/scope-expansion.md), profile
schema: [`shared/contracts/profile/schema.json`](../../../src/fdai/shared/contracts/profile/schema.json).

- **Upstream ships three canonical profiles**:
  - `baseline` - the minimum-safe posture, 10 rules, ships to any
    Azure tenant with zero customization.
  - `recommended` `extends: [baseline]` - standard best-practice; adds
    diagnostic settings, private endpoints, purge protection, RBAC,
    the full tag matrix. The current resolved total is 44 rules.
  - `strict` `extends: [recommended]` - regulated / zero-trust; moves
    security-critical rules from shadow to `enforce`. The current resolved total is 45 rules.
- **Upstream also ships 265 auto-imported profiles** under
  `rule-catalog/profiles/collected/` - one profile per Azure Policy
  built-in initiative (CIS Azure Foundations, NIST 800-53, PCI DSS,
  HIPAA HITRUST, ISO 27001, FedRAMP High / Moderate, GDPR, DORA,
  EU NIS2, CMMC, and every other regulatory framework Microsoft
  publishes as a policy set). Each references imported rules by their
  FDAI id.
- **Fork overrides** live under
  [`rule-catalog/profiles-overrides/`](../../../rule-catalog/profiles-overrides)
  (upstream empty). A fork adds a YAML with `extends: [strict]` and
  its own overrides, gives it a fork-owned id, and points the
  composition-root config at that id.

### Resolution

`ProfileRegistry.resolve(profile_id)` walks the `extends` DAG and
merges children over parents. The merge rules are:

- **Mode**: child override wins; missing = inherit; default = `shadow`.
- **Parameters**: shallow merge in order `profile.parameters` <-
  parent rule params <- child rule params.
- **Severity override**: child wins; a downgrade below the rule's
  authored floor raises `ProfileResolutionError` (fail-closed at
  load, not at runtime).
- **Disabled**: child `disabled: true` removes the rule from the
  resolved set.
- **Cycles** in the `extends` graph raise `ProfileResolutionError`.
- **Unknown parent** or **unknown rule id** (when `known_rule_ids` is
  supplied) raise `ProfileResolutionError`.

The resolved rule list is sorted by rule id so a diff between two
resolutions is byte-stable.

## 2. Collector pipeline

Every source is declared under
[`rule-catalog/sources/<id>/manifest.yaml`](../../../rule-catalog/sources)
per the shape in
[`source_manifest.schema.json`](../../../src/fdai/rule_catalog/schema/source_manifest.schema.json).
The `parser` field names one of the parser plugins registered under
[`src/fdai/rule_catalog/pipeline/parse/`](../../../src/fdai/rule_catalog/pipeline/parse).

### Shipped upstream collectors

| Source id | Origin | Parser | Rule count landed | Layout |
|-----------|--------|--------|-------------------:|--------|
| `fdai-p1-seed` | this repo | `rule-yaml` | 61 hand-authored | `rule-catalog/catalog/*.yaml` |
| `azure-policy-builtin` | `Azure/azure-policy` | `azure-policy-json` | 3628 | `rule-catalog/collected/azure-builtin/<Category>/*.yaml` |
| `kube-bench` | `aquasecurity/kube-bench` | `kube-bench` | 4859 | `rule-catalog/collected/kube-bench/<ruleset>/*.yaml` |
| `gatekeeper-library` | `open-policy-agent/gatekeeper-library` | `rego` | (schema-only; collector wiring pending) | `rule-catalog/collected/gatekeeper/*.yaml` |

### Reserved-but-unimplemented parsers

Declared in [`ParserName`](../../../src/fdai/rule_catalog/pipeline/parse/parser.py)
so a manifest referencing them fails at collect time with a clear
`ParserNotImplementedError`:

- `checkov-yaml` (Bridgecrew / Prisma Checkov IaC rulesets)
- `gatekeeper-templates` (OPA Gatekeeper `ConstraintTemplate` YAML)

Adding a new parser is a two-step change: register the class under
`build_parser` in `parser.py`, then add its manifest under
`rule-catalog/sources/`.

## 3. What "have all the policies" means

Upstream FDAI **does not** claim to hand-author every policy the
industry publishes. What it does claim:

1. **Every public reference framework can be imported**: the collector
   pipeline is source-agnostic and any well-formed JSON/YAML corpus
   can be added with one parser plugin.
2. **The imported reference IS shipped upstream** so a fork inherits
   the full corpus on day one - no external tooling required to see
   what rules exist.
3. **Curation is separate from collection**: imported rules stay
   `shadow`-only and point at the fail-closed
   `remediate.azure-policy-managed` ActionType until a curator writes
   real Rego. This is the safety invariant that makes it OK to import
   thousands of rules at once.

Coverage as of this document:

| Layer | Count |
|-------|------:|
| Hand-authored rules | 61 |
| Imported (Azure Policy built-in) | 3628 |
| Imported (kube-bench CIS Kubernetes) | 4859 |
| Profiles - upstream curated | 3 (`baseline`, `recommended`, `strict`) |
| Profiles - auto-imported compliance frameworks | 265 (CIS / NIST / HIPAA / PCI / ISO / FedRAMP / GDPR / DORA / ...) |

## 4. Fork adoption playbook

A fork's typical flow to activate a compliance posture:

```yaml
# rule-catalog/profiles-overrides/customer-a.yaml
id: customer-a
title: (Customer A) posture
extends:
  - strict                                              # upstream base
  - compliance.regulatory-compliance.cis-azure-foundations-v3-0-0
parameters:
  tag.mandatory: [Environment, Owner, CostCenter, Confidentiality]
rules:
  - id: azure-builtin.object-storage.storage-account-should-require-secure-transfer
    mode: enforce
  - id: azure-builtin.sql-server.deprecated-audit-sql-servers-with-auditing-enabled
    disabled: true                                      # customer-specific exemption
```

The target composition contract reads `FDAI_PROFILE_ID=customer-a` and hands the resolved profile
to `ControlLoop` / `T0Engine` / `RiskGate` at startup. The default composition doesn't implement
that contract yet.

> **Wiring status (2026-07):** the `ProfileRegistry` library
> (`src/fdai/core/rule_catalog_profiles/`) is shipped and covered by
> tests, but the composition root does not yet read `FDAI_PROFILE_ID`.
> The `resolve()` call must be added to
> [`src/fdai/composition/`](../../../src/fdai/composition/) before this
> knob has any runtime effect; fork maintainers who need the profile
> layer today can bind their own resolved profile via a wrapping factory
> until the upstream default binder wires it.
