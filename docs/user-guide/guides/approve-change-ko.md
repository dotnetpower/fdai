# 변경 승인(Approve a change)

제안된 변경이 HIL 티어로 오면 FDAI는 실행을 멈추고 사람의 결정을 기다립니다. 이
가이드에서는 요청의 구성, 승인 전 확인 항목, 각 판정 이후의 동작을 설명합니다.

## HIL 요청 구성

구성된 모든 알림 채널에서 승인 대기 사실을 받을 수 있습니다. 실제 승인 결정은 Teams
Adaptive Card, 재인증이 구성된 Slack 흐름, remediation PR 검토처럼 신원을 확인할 수
있는 A1 표면으로 반환해야 합니다. 이메일, SMS, 호출 시스템은 요청 대기 사실을 알릴 수
있지만 승인을 제출할 수 없습니다.

모든 HIL 요청은 표면과 관계없이 동일한 핵심 정보를 보여 줍니다:

- **이벤트 요약** - 변경을 트리거한 것(드리프트, 비용 이상, DR 훈련 등)과 영향을
  받는 리소스.
- **제안 액션** - FDAI가 적용하려는 정확한 변경. 검토 가능한 PR 또는
  직렬화된 액션 명세로 제공됩니다.
- **리스크 분류** - 왜 AUTO가 아니라 HIL인지의 이유. 티어를 올린 특정 차원(blast
  radius, 새로움, 되돌리기 어려움, 신호 소스).
- **롤백 미리보기** - 승인 후 되돌려야 할 때 실행될 사전 계산된 롤백 경로.
- **Stop-condition** - 승인 후 환경이 예상과 다르게 반응하면 변경을 멈출 측정 가능한
  상태.
- **감사 링크** - 이 결정을 만든 이벤트 체인을 볼 수 있는 감사 로그 딥링크.
- **승인 무결성** - 요청 TTL, 필요한 정족수, action hash, idempotency key, 요청자와
  승인자가 달라야 한다는 확인 결과.

## 승인 전 확인할 것

중요한 순서대로 여섯 가지를 확인하세요:

1. **리스크 분류가 맞는가?** 제안 액션이 명시된 리스크 대비 과도하게 느껴지면
  분류 규칙 자체를 검토해야 합니다. 그대로 승인하지 말고 거부한 뒤 담당자에게
  에스컬레이션하세요.
2. **Blast radius** - 범위 상한("이 리소스 그룹만", "VM 5개 배치" 등)이 실제로
  바꾸고 싶은 것과 맞는지 확인합니다.
3. **롤백 경로** - 롤백 미리보기는 내용이 있고 실제로 실행 가능해야 합니다. 비어 있거나
  모호한 롤백은 액션의 설계 결함이며 승인으로 우회할 수 없습니다.
4. **Stop-condition** - 이미 보고 있는 메트릭에서 관찰 가능해야 합니다. 볼 수 없는
  메트릭을 참조하면 거부하세요.
5. **Grounding (T2만)** - T2 결정이면 감사 로그 항목의 인용된 규칙과 문서가 제안
  액션을 실제로 뒷받침하는지 검증합니다.
6. **승인이 이 액션에 계속 연결되어 있는가?** 버전, 대상 범위, TTL, action hash,
  정족수를 확인합니다. 검토 후 페이로드가 바뀌었다면 새 승인 요청을 요구하세요.

## 판정과 그 결과

- **승인** - 신원, 역할, hash, TTL, 자기 승인 금지, 정족수 검사를 통과하면 저장된
  변경을 모든 안전 불변식(stop-condition, 롤백, blast-radius 상한, 감사 항목)과 함께
  재개합니다. 감사 로그는 누가 언제 승인했는지, 어떤 코멘트를
  남겼는지 기록합니다.
- **거부** - 변경은 폐기됩니다. 감사 항목은 여전히 기록(승인자, 이유, 이벤트 ID)
  되어 discovery loop가 패턴을 학습합니다.
- **시간 초과** - HIL 요청은 구성 가능한 타임아웃을 가집니다. 만료 시 변경은 거부와
  동일하게 폐기됩니다. **타임아웃 자동 승인은 절대 없습니다.**

중복 승인 제출은 idempotent하게 처리되며 실행을 replay할 수 없습니다. 충돌하는 응답은
거부하고 검토 대상으로 표시합니다.

## Break-glass 승인

BreakGlass는 별도의 시간 제한 긴급 역할입니다. 정책이 허용하는 긴급 HIL 승인에서 원래
자격이 없는 호출자에게 참여 자격을 부여할 수 있습니다. **DENY를 HIL로 바꾸거나 액션을
AUTO로 높이거나 정족수와 자기 승인 금지 검사를 제거하지 않습니다.**

모든 BreakGlass 사용은 grant, 사유, 승인자, 만료 시각, 영향받는 액션을 기록하고 온콜
팀에 알립니다. 운영자는 사후에 사유를 추가해야 합니다. BreakGlass는 근본 규칙이나 안전
계약을 수정하는 작업을 대신하지 않습니다.
